# RLS Faz 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 11 tabloda RLS açıp SELECT/INSERT/UPDATE'i serbest bırakarak, DELETE'i veritabanı seviyesinde tamamen engellemek — devtools'tan ham bir DELETE isteği bile artık işe yaramayacak.

**Architecture:** Tek bir SQL scripti (kullanıcı Supabase SQL editöründe çalıştırır), `begin`/`commit` ile atomik — 11 tabloya aynı 4 satırlık şablon uygulanır. Kod değişikliği yok. Doğrulama, Claude'un `curl` ile doğrudan bir DELETE isteği gönderip reddedildiğini teyit etmesiyle yapılır.

**Tech Stack:** Postgres RLS (Supabase). Kod tabanına dokunulmuyor.

## Global Constraints

- Her tabloda `SELECT`/`INSERT`/`UPDATE` politikaları `using(true)`/`with check(true)` ile TAMAMEN serbest bırakılır — bugünkü davranış birebir korunur, regresyon YOK.
- DELETE için hiçbir politika tanımlanmaz — Postgres RLS kuralı gereği bu, DELETE'i varsayılan olarak reddeder.
- `yevmiye_kalemleri`/`fatura_kalemleri`/`recete_kalemleri`/`siparis_kalemleri` gibi satır-kalemi tablolarına DOKUNULMAZ — onlar hâlâ sil-yeniden-yaz deseni kullanıyor, DELETE'e ihtiyaçları var.
- Tüm SQL tek bir `begin`/`commit` transaction'ı içinde — biri hata verirse hiçbiri uygulanmaz.
- Kapsam TAM OLARAK 11 tablo: `cariler`, `faturalar`, `demirbaslar`, `cek_senetler`, `banka_kasa_hesaplari`, `butce_kayitlari`, `kullanicilar`, `stok`, `stok_hareketleri`, `cari_hareketler`, `banka_kasa_hareketleri`.

### Task 2: Uçtan uca doğrulama

**Files:** (yok — sadece doğrulama)

- [ ] **Step 1: DELETE'in reddedildiğini doğrudan test et**

ÖNEMLİ (uygulama sırasında düzeltildi): var olmayan bir ID ile test etmek
YANILTICI — PostgREST, "RLS eşleşen satır bırakmadı" durumunda da "id zaten
yok" durumunda da AYNI yanıtı (`200`/`204`, 0 satır) döndürür, `403` DEĞİL.
Doğru test: GERÇEK bir satır oluştur, onu silmeye çalış, sonra hâlâ var
olup olmadığını kontrol et.

```bash
# Gerçek bir test satırı oluştur (butce_kayitlari, hesap_plani'ndan geçerli bir kod ile)
REAL_HESAP=$(curl -s "$SB_URL/rest/v1/hesap_plani?select=kod&limit=1" -H "apikey: $SB_KEY" -H "Authorization: Bearer $SB_KEY" | grep -o '"kod":"[^"]*"' | head -1 | cut -d'"' -f4)
INSERT_RESP=$(curl -s -X POST "$SB_URL/rest/v1/butce_kayitlari" -H "apikey: $SB_KEY" -H "Authorization: Bearer $SB_KEY" -H "Content-Type: application/json" -H "Prefer: return=representation" -d "{\"yil\":1900,\"otel_id\":\"810\",\"hesap_kodu\":\"$REAL_HESAP\"}")
TEST_ID=$(echo "$INSERT_RESP" | grep -o '"id":"[^"]*"' | head -1 | cut -d'"' -f4)

# Silmeyi dene
curl -s -w "\nHTTP:%{http_code}\n" -X DELETE "$SB_URL/rest/v1/butce_kayitlari?id=eq.$TEST_ID" -H "apikey: $SB_KEY" -H "Authorization: Bearer $SB_KEY" -H "Prefer: return=representation"

# Hâlâ duruyor mu kontrol et
curl -s "$SB_URL/rest/v1/butce_kayitlari?id=eq.$TEST_ID&select=id" -H "apikey: $SB_KEY" -H "Authorization: Bearer $SB_KEY"
```

(`$SB_URL`/`$SB_KEY` için `stok-takip.html`'deki `SB_URL`/`SB_KEY` sabit değerlerini kullan.)

Expected: DELETE isteği `200`/`204` dönebilir (bu normal — RLS hata kodu
değil, "0 satır etkilendi" ile engelliyor) AMA son SELECT'te satır
**hâlâ mevcut olmalı**. Satır kaybolmuşsa RLS çalışmıyor demektir, hemen
durup kullanıcıya bildir. Test satırını `PATCH {"silindi":true}` ile
temizle (bu, UPDATE'in de çalıştığını doğrular).
